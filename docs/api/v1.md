# Soccer Tournament Dashboard · API Reference · v1

## Overview

This document defines the v1 REST API for the Soccer Tournament Dashboard.  
All endpoints are prefixed with `/api/v1`. User-facing endpoints serve frontend page data. Admin endpoints are internal, protected by token auth, and called exclusively by background refresh jobs.

Tournament-scoped resources are nested under `/tournaments/{tournament_id}`. Independent resources such as teams, matches, search, health, and admin operations remain separate.

All error responses follow a standard format. See [Error Format](#error-format) below.

---

## User-Facing Endpoints

### Tournaments

Available tournaments and tournament details.

| Method | Endpoint                       | Description                            |
| ------ | ------------------------------ | -------------------------------------- |
| GET    | `/tournaments`                 | Available tournaments for the selector |
| GET    | `/tournaments/{tournament_id}` | Details for a single tournament        |

**Notes:**

- The tournament selector uses `/tournaments` to populate available competitions.
- Tournament-specific page data is exposed as child routes under `/tournaments/{tournament_id}`.
- This keeps standings, matches, and player stats grouped under the tournament that owns them.

---

### Tournament Standings

Group standings for a tournament. Each group contains its teams ranked by position.

| Method | Endpoint                                 | Description                                  |
| ------ | ---------------------------------------- | -------------------------------------------- |
| GET    | `/tournaments/{tournament_id}/standings` | All groups and ranked teams for a tournament |

**Query parameters:**

| Param   | Type   | Description                         |
| ------- | ------ | ----------------------------------- |
| `group` | string | Filter to a single group (e.g. `A`) |

**Notes:**

- Response groups teams by group label, ranked within each group by points, goal difference, and goals scored.
- In pre-tournament state, teams are returned with zeroed stats and no ranking applied.
- When `group` is provided, the endpoint returns the same grouped response shape, narrowed to one group.

---

### Tournament Matches

Match schedule for a tournament, with optional filters.

| Method | Endpoint                               | Description                  |
| ------ | -------------------------------------- | ---------------------------- |
| GET    | `/tournaments/{tournament_id}/matches` | All matches for a tournament |

**Query parameters:**

No query parameters currently supported.

**Notes:**

- Returns all matches for the selected tournament.
- Default ordering is `kickoff_time` ascending.
- The frontend groups matches by date.
- Pagination is not used because tournament fixture lists are bounded and small.

---

### Tournament Player Stats

Tournament leaderboard, sortable by stat category.

| Method | Endpoint                                    | Description                               |
| ------ | ------------------------------------------- | ----------------------------------------- |
| GET    | `/tournaments/{tournament_id}/player-stats` | Player stats leaderboard for a tournament |

**Query parameters:**

| Param   | Type    | Description                                                                                       |
| ------- | ------- | ------------------------------------------------------------------------------------------------- |
| `stat`  | string  | Stat to sort by: `goals`, `assists`, `yellow_cards`, `red_cards`, `appearances`, `minutes_played` |
| `limit` | integer | Number of players to return (default: 10)                                                         |

**Notes:**

- Results are ordered by `stat` descending.
- Tiebreaker is `minutes_played` ascending.
- Sorting is handled server-side so leaderboard results are correct across the full dataset.

---

### Teams

Team details and team-specific tournament data.

| Method | Endpoint                   | Description                                          |
| ------ | -------------------------- | ---------------------------------------------------- |
| GET    | `/teams/{team_id}`         | Team details (name, short name, type, country, logo) |
| GET    | `/teams/{team_id}/roster`  | Squad registered for a tournament                    |
| GET    | `/teams/{team_id}/matches` | Matches involving this team                          |

**Query parameters for `/teams/{team_id}/roster`:**

| Param           | Type    | Description                               |
| --------------- | ------- | ----------------------------------------- |
| `tournament_id` | integer | Required, roster is scoped per tournament |

**Query parameters for `/teams/{team_id}/matches`:**

| Param           | Type    | Description                                |
| --------------- | ------- | ------------------------------------------ |
| `tournament_id` | integer | Optional, narrows to a specific tournament |
| `status`        | string  | Filter by match status (e.g. `scheduled`)  |
| `limit`         | integer | Limit number of results                    |

**Notes:**

- Teams are independent resources because they can be viewed outside a specific tournament page.
- `tournament_id` is required for roster because squad numbers and positions are registration attributes scoped per tournament.
- The Team Profile page can request team details, roster, and matches independently.

---

### Match Details

Single-match details and event timeline.

| Method | Endpoint                     | Description                       |
| ------ | ---------------------------- | --------------------------------- |
| GET    | `/matches/{match_id}`        | Details for a single match        |
| GET    | `/matches/{match_id}/events` | Event timeline for a single match |

**Notes:**

- Match details are independent because a match has its own identity after it is selected.
- `/matches/{match_id}/events` returns events ordered chronologically.
- Events include goals, assists, cards, substitutions, and penalty events.

---

### Search

Unified search across teams and players.

| Method | Endpoint  | Description                         |
| ------ | --------- | ----------------------------------- |
| GET    | `/search` | Search teams and/or players by name |

**Query parameters:**

| Param  | Type   | Description                              |
| ------ | ------ | ---------------------------------------- |
| `q`    | string | Search term (e.g. `messi`, `argentina`)  |
| `type` | string | Scope: `all` (default), `team`, `player` |

**Response shape:**

```json
{
    "teams": [],
    "players": []
}
```

**Notes:**

- When `type=team`, `players` is an empty array.
- When `type=player`, `teams` is an empty array.
- When `type=all`, both are populated.

## Admin Endpoints

All admin endpoints are under `/api/v1/admin` and require a bearer token set via the `ADMIN_TOKEN` environment variable. Unauthenticated requests return `403`.

Admin endpoints are internal operational endpoints. They are used for manual maintenance, scheduled refresh triggers, and controlled writes from trusted tooling. They are not user-facing.

---

### Manual Write Operations

| Method | Endpoint                                       | Description                         |
| ------ | ---------------------------------------------- | ----------------------------------- |
| POST   | `/admin/tournaments`                           | Insert tournament                   |
| POST   | `/admin/teams`                                 | Insert team                         |
| POST   | `/admin/players`                               | Insert player                       |
| POST   | `/admin/matches`                               | Insert match                        |
| POST   | `/admin/standings`                             | Insert standings row                |
| POST   | `/admin/player-stats`                          | Insert player stats row             |
| POST   | `/admin/match-events`                          | Insert match event                  |
| PUT    | `/admin/matches/{match_id}`                    | Update score and status for a match |
| PUT    | `/admin/tournaments/{tournament_id}/standings` | Upsert standings for a tournament   |
| PUT    | `/admin/tournaments/{tournament_id}/matches`   | Upsert matches for a tournament     |

**Notes:**

- No `DELETE` endpoints. API-Football data is corrected via `PUT`, not deleted.
- Tournament refresh `PUT` endpoints receive already-transformed request bodies.
- These endpoints are useful for manual debugging, scripts, and controlled data correction.

---

### HTTP Refresh Triggers

| Method | Endpoint                               | Description                                   |
| ------ | -------------------------------------- | --------------------------------------------- |
| POST   | `/admin/tournaments/refresh-standings` | Fetch and refresh standings from API-Football |
| POST   | `/admin/tournaments/refresh-matches`   | Fetch and refresh matches from API-Football   |

**Query parameters:**

| Param         | Type    | Default                             | Description                                       |
| ------------- | ------- | ----------------------------------- | ------------------------------------------------- |
| `margin_days` | integer | `1` for standings, `14` for matches | Number of days around tournament dates to include |

**Refresh flow:**

1. Backend finds refreshable tournaments.
2. Backend calls API-Football using the server-side API key.
3. External responses are transformed into internal schemas.
4. Existing update services are called.
5. Database rows are upserted.
6. Relevant cache entries are invalidated.
7. A refresh summary is returned.

**Example response:**

```json
{
    "message": "Matches refresh completed",
    "resource_name": "Matches",
    "tournaments_checked": 2,
    "tournaments_refreshed": 2,
    "tournaments_skipped": 0,
    "rows_processed": 128,
    "failures": []
}
```

If one tournament fails, the refresh continues for remaining tournaments:

```json
{
    "message": "Standings refresh completed with failures",
    "resource_name": "Standings",
    "tournaments_checked": 2,
    "tournaments_refreshed": 1,
    "tournaments_skipped": 0,
    "rows_processed": 32,
    "failures": [
        {
            "tournament_id": 4,
            "external_api_id": 9,
            "season": "2024",
            "reason": "API-Football request failed"
        }
    ]
}
```

**Notes**:

- These endpoints are intended for external schedulers such as cron-job.org.
- The scheduler only sends authenticated HTTP requests.
- API-Football credentials stay inside the backend.
- The backend owns refresh orchestration, transformation, writes, and cache invalidation.

## Error Format

All errors return a consistent JSON shape:

```json
{
    "error": {
        "status": 404,
        "code": "NOT_FOUND",
        "message": "Tournament not found"
    }
}
```

| Status | Code             | When                                    |
| ------ | ---------------- | --------------------------------------- |
| 400    | `BAD_REQUEST`    | Invalid query parameter or request body |
| 403    | `UNAUTHORIZED`   | Missing or invalid admin token          |
| 404    | `NOT_FOUND`      | Resource does not exist                 |
| 409    | `CONFLICT`       | Resource already exists                 |
| 429    | `RATE_LIMITED`   | Too many requests                       |
| 500    | `INTERNAL_ERROR` | Unhandled server error                  |

**Implementation:** Custom exception classes (`NotFoundError`, `UnauthorizedError`, etc.) inherit from a base `AppError` with a `status_code` attribute. A single FastAPI exception handler catches any `AppError` subclass and packages it into this format. Service layer raises typed exceptions, no string-to-status-code mapping.

## Architecture

Requests follow a strict layered flow:

```txt
Router → Service → Repository → Database
```

| Layer      | Folder          | Responsibility                                                                           |
| ---------- | --------------- | ---------------------------------------------------------------------------------------- |
| Router     | `routers/`      | HTTP handling, param extraction, input validation via Pydantic schemas, response shaping |
| Client     | `clients/`      | External API communication, request headers, timeouts, response retrieval                |
| Service    | `services/`     | Business logic, orchestration, tiebreaker rules, multi-repository calls                  |
| Repository | `repositories/` | Database access only, SQLAlchemy queries, no business logic                              |
| Models     | `models/`       | SQLAlchemy table definitions                                                             |
| Schemas    | `schemas/`      | Pydantic request/response shapes, decoupled from models                                  |
